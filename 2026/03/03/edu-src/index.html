EDU SRC 挖洞实战：我常用的小程序打法

发表于2026-03-03|更新于2026-03-03|红队攻防web安全

最近一直被催着分享之前挖 CNVD 的思路，这篇就做一次系统化复盘：我在 EDU SRC 里常用的小程序打法。

说明：文中截图均来自真实案例，相关目标均已修复，内容仅用于授权测试场景下的技术交流。

1. 总体打法：先小程序，再业务链

很多人会优先打 Top10、绕 WAF、做社工，这些方向都没问题。
但从“效率和稳定出洞率”看，我更推荐这条路线：
小程序入口 -> 身份凭据/JWT -> 业务接口联动 -> 敏感数据与权限问题。

这条链路的核心价值是：接口集中、业务闭环明显、可快速形成高质量复现链。

2. 常见框架的系统化切入

先做框架识别，再做针对性测试。
在 EDU 场景里，若依、Jeecg、Admin.NET、Spring Boot 出现频率很高。

2.1 若依（Ruoyi）

常见特征路径：/prod-api、/xxx-api。

重点切入点：

  1. 默认口令/弱口令（例如 ruoyi/admin123、ry/admin123）

一旦进入后台，再看计时任务、已知 Nday、前台 Shiro 等可利用面，容易串成完整风险链。

  1. 小程序一键登录拿到 JWT 后，直接验证敏感接口

拿到身份证号、手机号等批量敏感信息，通常就具备高危定级条件。
若依中也常见“获取初始密码”这类高风险接口。

  1. 没有 Session 时，补测 Druid 弱口令

2.2 Jeecg

常见路径：/jeecg-boot/、/xxxx-boot/。

重点切入点：

  1. 登录拿 JWT 后跑敏感接口
  2. 测试默认口令/弱口令（例如 jeecg/123456）
  3. 后台排查积木报表等 SQL 注入风险

此外，部分系统会存在普通用户提权到管理员的接口，值得重点梳理。

2.3 Admin.NET

常见接口前缀：/api/sysxxxxxx。
打法与前面类似，拿到凭据后按业务域批量验证，经常能发现敏感证书或配置文件泄露。

2.4 Spring Boot

重点切入点：

  1. 先看经典报错页面

  1. 再看 Swagger 暴露情况

接口逐个验证时，未授权、越权、字段越权都可能出现。
测试时要控制范围，避免触发删除或破坏性操作。

3. 小程序场景下的高频得分点

我平时不少分数都来自业务逻辑漏洞，尤其是“先拿登录态，再横向点功能”的打法。

3.1 登录逻辑漏洞

重点观察点：

  1. 登录请求是否泄露 sessionKey，用于解密数据或伪造登录态

  1. 越权 + 登录逻辑组合，可形成全站用户接管

当越权返回包含 openid、unionid、手机号等字段时，继续联动登录接口验证。

如果登录仅校验“手机号 + openid + unionid”三要素，拿到三项即可登录他人账号。

3.2 登录后的头像上传

这个点常被低估。上传校验一旦松动，后续利用空间会迅速放大。

建议先做可控回显验证，再按授权范围逐步推进。

3.3 验证码绕过

重点切入点：

  1. 验证码和手机号都在请求包里可改

将手机号改为自己的号码，再填自己收到的验证码，经常可直接绕过。

  1. 四位验证码可被直接爆破

3.4 手机号爆破

如果接口缺少频控和风控策略，风险很容易放大。

4. 复现与提交流程建议

为了让漏洞报告更稳，建议按这个模板组织复现：

  1. 入口与前置条件：从哪里进、需要什么身份
  2. 复现步骤：每一步请求与关键参数变化
  3. 结果证明：返回数据、权限变化、影响范围
  4. 风险说明：可造成的数据泄露或业务影响
  5. 修复建议：鉴权补齐、频控策略、字段最小化返回等

5. 总结

这篇文章复盘的是我在 EDU SRC 里一条稳定的实战思路：
小程序入口 -> 身份凭据/JWT -> 业务接口联动 -> 敏感数据与权限问题。

实战中多做“接口之间的关联验证”，通常比单点盲测更容易拿到高质量漏洞。
