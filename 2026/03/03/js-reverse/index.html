记一次小程序挖掘：JS 逆向请求与响应

发表于2026-03-03|更新于2026-03-03|红队攻防web安全

这篇记录一次 EDU 小程序接口逆向过程，重点是把请求与响应的加解密链路完整还原出来。
文中目标均已修复，仅用于授权测试场景下的技术复盘。

1. 问题现象

抓包后发现请求与响应都不是明文，核心参数都做了加密处理，无法直接在 Burp 中复现业务接口。

2. 请求链路逆向

2.1 定位加密入口

反编译小程序并导入微信开发者工具后，全局搜索 requestData，定位到请求封装位置。
这里同时出现了 requestData 和 iv 的生成逻辑。

2.2 拆解 requestData 参数来源

断点后可以看到：requestData 最终由 t() 函数处理，而 t() 之前有三个关键输入值。

第一个参数 c：
JSON.stringify(c) 多次观察后可确认是业务请求载荷，字段如 class、page 控制模块与分页。

第二个参数 u：
向上跟踪可见由 d() 生成。

进入 d() 后可确认其本质是随机字符串生成函数。

第三个参数 a.decodekey：
调试时为 undefined，在当前链路下可先忽略。

2.3 还原加密函数 t()

继续分析包裹函数 t()，可确认是 AES-CBC + Pkcs7。

调试结果显示：

  1. e 为待加密明文（业务 JSON 字符串）
  2. 第二参数 n 即上文 u，用于生成 iv
  3. key 为固定字符串：WAG0JIUGYALACVJF

到这里请求端逻辑已经打通：只要本地复刻同样的 key、iv 与加密模式，就能生成有效 requestData。

3. Node.js 复刻请求加密

运行后可得到可用的 requestData 与 iv。

4. Burp 验证请求可用性

将生成值填入请求后转发，接口能正常返回业务数据，说明请求加密复刻成功。

5. 响应链路逆向

接下来处理响应解密。全局搜索 decrypt，定位到解密函数。

调试后可确认：

  1. e 是响应密文（Base64）
  2. n 是固定 iv：XV9LXBQGYXPIZHA7
  3. key 仍是固定字符串：WAG0JIUGYALACVJF

6. Node.js 复刻响应解密

运行后可直接还原响应明文，拿到业务数据与敏感资源链接。

7. 总结

这次核心不在于“算法多复杂”，而在于把链路走通：
定位参数 -> 追踪函数 -> 断点确认 key/iv -> 本地复刻 -> Burp 验证 -> 反向解密响应。

在小程序场景里，前端加密只要密钥和流程落在客户端，最终都可能被还原。
做 SRC 时，建议优先关注这种“可复刻通信链路 + 敏感数据返回”的组合点。
